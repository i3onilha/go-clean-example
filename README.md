# Go Clean Architecture Example

A production-ready Go web application demonstrating **Clean Architecture** principles with Gin framework, MySQL database, and dependency injection using Uber Fx.

## Architecture

This project follows Clean Architecture with clearly separated layers:

```
internal/
├── controller/    # HTTP handlers (delivery layer)
├── usecase/       # Application business logic
├── domain/        # Enterprise business rules & entities
├── dto/           # Data transfer objects
└── repository/    # Data access layer (MySQL with sqlc)
```

### Key Principles

- **Domain layer** contains core business entities and rules that must never depend on outer layers
- **Usecase layer** orchestrates business logic and coordinates repositories
- **Controller layer** handles HTTP requests/responses and delegates to usecases
- **Repository layer** manages data persistence with type-safe SQL generated by sqlc

## Tech Stack

- **Go 1.25** - Programming language
- **Gin** - HTTP web framework
- **Uber Fx** - Dependency injection
- **MySQL 8.0** - Database
- **sqlc** - Type-safe SQL code generation
- **Viper** - Configuration management

## Getting Started

### 1. Clone the Repository

```bash
git clone <repository-url>
cd go-clean-example
```

### 2. Configure Environment

Copy the example environment file and adjust as needed:

```bash
cp .env.example .env
```

Default environment variables:

| Variable | Description | Default |
|----------|-------------|---------|
| `APP_ENV` | Application environment | `development` |
| `REST_PORT` | HTTP server port | `3000` |
| `APP_VERSION` | Application version | `1.0` |
| `MYSQL_USER` | Database user | `default` |
| `MYSQL_PASSWORD` | Database password | `secret` |
| `MYSQL_DATABASE` | Database name | `erp_sales` |
| `MYSQL_DSN` | Database connection string | - |

### 3. Run with Docker Compose

Start the application and MySQL database:

```bash
docker-compose up -d
```

The services will be available at:
- **API**: `http://localhost:8080`
- **GoDoc**: `http://localhost:6464`
- **MySQL**: `localhost:3306`

### 4. Run Locally (Development)

Install dependencies:

```bash
go mod download
```

Install development tools:

```bash
go install github.com/air-verse/air@latest
```

Run with live reload:

```bash
air
```

## Code Generation

### Generate Type-Safe SQL Code

After modifying SQL queries or schema:

```bash
sqlc generate
```

This generates type-safe Go code from SQL queries defined in `internal/repository/mysql/`.

## API Endpoints

### Get User with Orders

```
GET /users/:id
```

**Query Parameters:**
- `limit` (optional): Number of orders to return (default: 20, max: 100)
- `offset` (optional): Pagination offset (default: 0)

**Example:**

```bash
curl http://localhost:8080/users/1?limit=10&offset=0
```

**Response:**

```json
{
  "user_id": 1,
  "name": "John Doe",
  "email": "john@example.com",
  "location": "New York",
  "orders": [
    {
      "order_id": 1,
      "user_id": 1,
      "item": "Widget",
      "quantity": 2,
      "price": 29.99,
      "discount_percent": 10,
      "final_price": 53.98
    }
  ]
}
```

## Business Logic

The domain layer contains critical business rules, such as order price calculation with discount validation:

- Discounts are clamped between 0% and 100%
- Final price = quantity × price × (1 - discount/100)

This logic lives in the domain layer and is never reimplemented elsewhere.

## Configuration

Configuration is managed via Viper with support for:

- `.env` file
- Environment variables (highest priority)
- Automatic environment variable mapping

Thread-safe singleton pattern ensures configuration is loaded once and accessed safely across goroutines.

## Development

### Live Reload

The project uses [air](https://github.com/air-verse/air) for hot reloading during development. Configuration is in `.air.toml`.

### Database Migrations

Schema definitions are located in `internal/repository/mysql/schema.sql`. Update this file and regenerate sqlc code.
